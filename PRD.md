# Tunnel Smart Advisor PRD

## 1. 문서 목적

Tunnel Smart Advisor는 터널 시공 조건과 현장 지식 데이터를 기반으로 공사 전·설계변경 검토 단계에서 잠재 리스크 누락을 줄이고, 관련 대응전략과 기준 근거를 빠르게 탐색하도록 돕는 내부 PoC 제품이다.

이 PRD는 현재 저장소에 구현된 FastAPI 백엔드, React/Vite 프론트엔드, CSV 온톨로지 파이프라인을 기준선으로 삼고, 장기 제품 방향을 P0/P1/P2 우선순위로 정리한다.

## 2. 제품 요약

- **제품명:** Tunnel Smart Advisor
- **대상 사용자:** 현장 시공 엔지니어, 본사 기술팀
- **사용 시점:** 착공 전 리스크 검토, 설계변경 검토
- **제품 단계:** 내부 PoC
- **핵심 목표:** 위험 누락 감소, 터널 시공 지식검색
- **배포 환경:** 사내 서버
- **인증 방식:** 사용자 로그인 및 API Key 없이 사내망에서 접근

## 3. 문제 정의

터널 공사는 지반, 공법, 장비, 위치, 공정 조건의 조합에 따라 위험이 크게 달라진다. 현장 엔지니어와 본사 기술팀은 착공 전 또는 설계변경 검토 시 과거 사례, 표준 체크리스트, 대응전략, KDS/KCS 기준 근거를 함께 검토해야 하지만, 현재는 관련 정보가 Excel, 문서, 개별 경험에 분산되어 있어 중요한 리스크를 누락하기 쉽다.

이 제품은 터널 시공 조건과 자연어 질의를 입력받아 관련 리스크, 근거 노드, 대응전략, 기준 조항을 연결해 보여주는 의사결정 지원 도구가 되어야 한다.

## 4. 성공 기준

### 4.1 제품 성공 기준

- 착공 전·설계변경 검토 시 주요 리스크 후보를 빠르게 식별한다.
- 현장 조건과 관련된 과거 리스크·대응전략·표준 기준을 한 화면에서 탐색한다.
- 분석 결과를 내부 검토, 보고자료, 발주처 제출용 리포트로 재사용한다.
- 원천 Excel 교체 후 데이터 버전과 갱신 상태를 추적할 수 있다.

### 4.2 PoC 검증 기준

- 현장 조건 입력 후 상위 리스크와 대응전략이 5초 이내 반환된다.
- 사용자가 리스크 카드 또는 그래프 노드를 클릭해 원문 LL 내용, 원인, 영향, 대응전략, 관련 프로젝트, 표준 근거, 담당 주체를 확인할 수 있다.
- 분석 결과가 이력과 리포트로 저장되고 재실행 가능하다.
- 터널 KDS/KCS 기준 조항을 리스크·전략 상세 화면에서 근거로 조회할 수 있다.

## 5. 현재 구현 기준선

### 5.1 백엔드

현재 FastAPI 백엔드는 다음 기능을 제공한다.

- `POST /score/`: 공종, 지반, 위치, 공법, 장비, 자연어 질의를 기반으로 리스크 랭킹, 대응전략, 지식그래프 JSON을 반환한다.
- `GET /nodes/{node_type}`: `process`, `ground`, `location`, `method`, `equipment`, `risk`, `strategy` 노드 목록을 반환한다.
- `GET /dashboard/summary`: 대시보드 KPI와 분포 데이터를 반환한다.
- `GET /library/items`: 리스크 중심 지식 라이브러리 목록을 반환한다.
- `GET /history/analyses`, `POST /history/analyses/{id}/rerun`: 분석 이력 조회와 최신 데이터 기준 재실행을 지원한다.
- `GET /reports`, `GET /reports/{id}.html`, `GET /reports/{id}.pdf`, `POST /reports/{id}/share`: 분석 결과 기반 리포트 조회, 다운로드, 공유 상태 관리를 지원한다.
- `GET /notifications`, `POST /notifications/*`: 알림 조회, 읽음, 중요 표시를 지원한다.
- `POST /admin/cache/reload`: CSV 캐시 재로딩을 지원한다.

### 5.2 프론트엔드

현재 React/Vite 프론트엔드는 다음 화면을 제공한다.

- 대시보드
- 분석 워크스페이스
- 지식 라이브러리
- 과거 분석
- 리포트
- 알림
- 설정 placeholder
- 도움말 placeholder

분석 워크스페이스는 5개 드롭다운 조건과 자연어 검색어를 입력받고, 리스크 카드, 종합 위험도 게이지, 권장 대응전략, vis-network 기반 지식그래프를 표시한다.

### 5.3 데이터/온톨로지

현재 런타임 데이터는 `data/tunnel_checklist/`의 CSV를 사용한다.

- 주요 노드: `Process`, `Risk`, `Strategy`, `Ground`, `Method`, `Equipment`, `Location`
- 추가 CSV 존재 노드: `Role`, `Standard`, `Impact`
- 주요 관계: `ENCOUNTERS`, `TRIGGER`, `OCCURS_AT`, `ASSOCIATED_WITH`, `MITIGATED_BY`, `REQUIRES`, `ASSIGNED_TO`, `BASED_ON`, `AFFECTS`
- 원천 데이터: `터널(NATM)표춘체크리스(26년4월 13일) (2).xlsx`
- 표준 파이프라인 목표: `scripts/ontology/build_master_ontology.py`를 canonical builder로 지정

## 6. 핵심 사용자 시나리오

### 6.1 착공 전 리스크 검토

1. 현장 엔지니어가 공종, 지반, 위치, 공법, 장비 조건을 선택한다.
2. 필요한 경우 자연어로 현장 맥락을 추가한다.
3. 시스템은 관련 리스크를 신뢰도 있는 점수와 등급으로 제시한다.
4. 사용자는 리스크별 원인, 영향, 과거 프로젝트, 대응전략, KDS/KCS 기준 근거를 확인한다.
5. 분석 결과를 저장하고 리포트로 출력한다.

### 6.2 설계변경 검토

1. 본사 기술팀이 변경 전·변경 후 조건을 비교한다.
2. 시스템은 새로 증가하거나 사라진 리스크, 영향도 변화, 추가 대응전략을 표시한다.
3. 변경 검토 결과를 내부 보고자료 또는 발주처 제출용 리포트로 생성한다.

### 6.3 지식검색

1. 사용자가 지식 라이브러리에서 리스크, 대응전략, 기준, 프로젝트 사례를 검색한다.
2. 검색 결과에서 상세 페이지를 열어 관련 그래프와 원문 근거를 확인한다.
3. 필요한 경우 관리자 권한으로 신규 지식 항목을 등록한다.

## 7. 기능 요구사항

### 7.1 P0: PoC 핵심 완성

#### P0-1. PRD 기준선 정리

- 현재 구현 기능과 목표 기능을 문서화한다.
- 루트 `PRD.md`를 제품 요구사항의 기준 문서로 사용한다.

#### P0-2. 분석 워크스페이스 고도화

- 위험도 필터 체크박스는 실제 리스크 목록 필터로 동작해야 한다.
- `조건 저장` 기능을 활성화해야 한다.
  - 저장 대상: 필터 조건과 자연어 질의문
  - 저장 범위: 전역 저장, 사내 접속 사용자 전체가 접근 가능
- 북마크는 현 상태처럼 세션/브라우저 상태 수준으로 유지한다.

#### P0-3. 프로젝트 범위 정책

- 프로젝트명 입력은 필수로 두지 않는다.
- 분석은 전체 프로젝트 데이터 기준으로 수행한다.
- 리스크 상세에는 `source_project`를 근거 정보로 표시한다.

#### P0-4. 지식그래프 상세 탐색

- 그래프 노드 클릭 시 상세 패널을 열어야 한다.
- 상세 패널은 다음 정보를 표시해야 한다.
  - 원문 LL 내용
  - 원인
  - 영향
  - 대응전략
  - 관련 프로젝트
  - 표준 근거
  - 담당 주체
- 그래프는 단순 시각화가 아니라 핵심 탐색 UI로 취급한다.

#### P0-5. `Role`, `Standard`, `Impact` 노출

- 현재 CSV에 존재하는 `Role`, `Standard`, `Impact` 노드를 API와 UI에서 조회 가능하게 한다.
- `Impact`는 리스크 상세, 필터, 대시보드 지표에 활용한다.
- `Standard`는 KDS/KCS 기준 근거 연결에 활용한다.
- `Role`은 우선 담당 주체 후보로 표시하되, 책임 워크플로는 P1 이후로 둔다.

#### P0-6. KCSC Standards 연동

- 사용자가 리스크 또는 대응전략 상세에서 관련 KDS/KCS 기준을 조회할 수 있어야 한다.
- KCSC Standards MCP를 사용해 기준 코드 검색, 조항 검색, 인용 검증을 수행한다.
- 우선 대상 기준 예시:
  - `KDS 27 00 00` 터널설계기준
  - `KCS 27 00 00` 터널공사
  - `KCS 27 20 00` 터널 굴착공사
  - `KCS 27 30 00` 터널 지보재 시공
  - `KCS 27 50 05` 터널 배수 및 방수 공사
- 상세 화면에는 기준명, 코드, 조항 경로, 원문 일부, 출처 URL, 기준 버전을 표시한다.

#### P0-7. 데이터 Source of Truth 정리

- `터널(NATM)표준체크리스트 Excel`을 공식 원천 데이터로 둔다.
- `build_master_ontology.py`를 canonical ontology builder로 지정한다.
- 다른 생성 스크립트는 실험/보조 스크립트로 분류하거나 canonical builder로 통합한다.
- 수동 Excel 교체 후 ontology rebuild, schema validation, smoke test, cache reload 순서를 표준 운영 절차로 문서화한다.

#### P0-8. 데이터 버전 표시

- 분석 결과와 리포트에 원천 Excel 버전 또는 데이터 빌드 버전을 표시한다.
- 예: `2026-04-13 체크리스트 기반`, `ontology_build_at`, `source_file_hash`.

### 7.2 P1: 신뢰도 있는 리스크 모델과 검토 워크플로

#### P1-1. 리스크 점수 모델 재설계

현재 점수 모델은 그래프 연결도 기반의 placeholder로 간주한다. P1에서는 다음 입력을 반영한 신뢰성 있는 점수 모델을 개발해야 한다.

- 발생 가능성
- 영향도
- 신뢰도
- 빈도
- 프로젝트 유사도
- 최신성
- 전문가 가중치

점수는 단순 연결 수가 아니라 통계적 의미를 갖는 값이어야 하며, 점수 산정 근거를 사용자에게 설명할 수 있어야 한다.

#### P1-2. 클러스터링 기반 위험 등급

- 현재 percentile 기반 `최상위/상위/중위험/저위험` 분류를 대체한다.
- 점수 분포를 통계 분석하고 의미 있는 클러스터가 형성되는 기준으로 위험 등급을 구분한다.
- 후보 방식:
  - K-means 또는 hierarchical clustering
  - 점수 분포 gap analysis
  - 이상치 탐지 기반 최상위 위험 분류
- 등급 경계와 모델 버전을 리포트에 기록한다.

#### P1-3. 리스크 데이터 스키마 확장

`Risk` 노드는 최소 다음 속성을 지원해야 한다.

- `cause`
- `impact`
- `likelihood`
- `impact_score`
- `frequency`
- `recency`
- `confidence`
- `expert_weight`
- `source_project`
- `source_version`

`Strategy` 노드는 최소 다음 속성을 지원해야 한다.

- `action`
- `target_risk`
- `expected_effect`
- `required_equipment`
- `related_standard`
- `responsible_role`

#### P1-4. 설계변경 검토 워크플로

- 변경 전 조건과 변경 후 조건을 비교하는 분석 모드를 추가한다.
- 결과는 다음 항목을 포함해야 한다.
  - 신규 발생 가능 리스크
  - 감소 또는 제거된 리스크
  - 위험등급 상승 리스크
  - 추가 대응전략
  - 관련 기준 근거
- 리포트는 단일 분석 스냅샷이 아니라 비교 검토 결과를 포함해야 한다.

#### P1-5. 리스크 누락 감지

- 선택한 공법/지반/위치 조합에서 일반적으로 함께 검토해야 하는 조건이나 리스크를 추천한다.
- 예: NATM + 파쇄대 + 도심지 조건에서 사용자가 지하수 조건을 선택하지 않은 경우 관련 누락 가능성을 알린다.
- 누락 감지는 단순 오류가 아니라 “추가 검토 권고”로 표현한다.

#### P1-6. 리포트 품질 개선

리포트는 내부 검토, 보고자료, 발주처 제출에 사용할 수 있어야 한다.

필수 항목:

- 입력 조건
- 상위 리스크
- 점수 산정 근거
- 대응전략
- 그래프 이미지 또는 관계 요약
- 관련 기준 조항
- 담당자/담당 주체
- 작성일
- 데이터 버전
- 모델 버전

#### P1-7. 알림 고도화

다음 이벤트에서 알림을 생성한다.

- 데이터 업데이트
- 리포트 공유
- 시스템 오류
- 분석 완료
- 고위험 결과 발생

알림은 읽음, 중요 표시, 삭제, 보관을 지원한다.

### 7.3 P2: 장기 제품화

#### P2-1. 지식 라이브러리 상세/등록

- 라이브러리 항목 상세 페이지를 제공한다.
- 관리자만 신규 지식을 등록할 수 있다.
- 등록 대상:
  - 리스크
  - 대응전략
  - Lessons Learned
  - 프로젝트 사례
  - 기준 문서
  - 장비/공법 정보
- 등록된 지식은 검증 상태와 데이터 버전을 가져야 한다.

#### P2-2. 철도/지하철 그래프 통합

- `data/neo4j_import/`의 철도/지하철 프로젝트·리스크·Lessons Learned 데이터를 본 지식그래프에 통합한다.
- 현재 목표 모델은 프로젝트별 별도 Risk/Strategy가 아니라 동일 리스크/전략을 통합하고 프로젝트를 속성 또는 관계로 연결하는 방식이다.
- 목표 구조:
  - 표준 Risk/Strategy 노드
  - Project 노드
  - Project ↔ Risk 사례 관계
  - Project ↔ Strategy 적용 관계

#### P2-3. 검색 고도화

- MVP에서는 키워드 기반 검색을 유지한다.
- 장기적으로는 백엔드 full-text search, 태그 검색, 관계 기반 검색을 지원한다.
- 의미검색/임베딩/RAG는 이번 PRD의 필수 범위는 아니지만, 향후 확장 후보로 둔다.

#### P2-4. 공유 기능 고도화

- 공유 링크 생성
- 사용자 초대 없는 링크 기반 공유
- PDF 다운로드
- 외부 제출용 리포트 패키지 생성

#### P2-5. 운영 대시보드

- 총 리스크 수
- 최상위 위험 수
- 데이터 최신성
- 최근 데이터 업데이트 상태
- 시스템 오류 상태
- 리포트 공유 현황

## 8. 비기능 요구사항

### 8.1 성능

- 기본 분석 응답은 사내 서버 기준 5초 이내를 목표로 한다.
- 그래프 렌더링은 상위 리스크 중심으로 제한해 브라우저 성능 저하를 방지한다.

### 8.2 보안

- 내부 PoC 단계에서는 사용자 로그인과 API Key를 모두 요구하지 않는다.
- 접근 통제는 사내 서버/사내망 운영 정책으로 처리한다.
- 관리자 기능은 별도 인증 기능을 만들지 않고 운영자 전용 절차와 서버 접근 권한으로 제한한다.

### 8.3 운영

- 사내 서버 배포를 기준으로 한다.
- 원천 Excel 교체는 수동 운영 절차로 수행한다.
- 데이터 갱신 후 validation, smoke test, cache reload를 수행한다.

### 8.4 감사성

- 분석 결과에는 데이터 버전과 모델 버전을 남긴다.
- 리포트 공유와 데이터 업데이트 이벤트는 알림과 로그로 추적한다.

## 10. 테스트 요구사항

### 10.1 P0 테스트

- `/score/`가 조건과 자연어 질의에 대해 리스크, 그래프, 이력 ID를 반환하는지 검증한다.
- `Role`, `Standard`, `Impact` 노드가 API에서 조회되는지 검증한다.
- 그래프 노드 클릭 상세 패널이 필수 정보를 표시하는지 프론트엔드 테스트를 작성한다.
- `조건 저장`과 위험도 필터 동작을 테스트한다.
- KCSC Standards 연동 서비스가 터널 기준 코드와 조항 검색 결과를 정상 반환하는지 테스트한다.

### 10.2 P1 테스트

- 신규 점수 모델이 동일 입력에 대해 재현 가능한 결과를 반환하는지 검증한다.
- 점수 산정 근거와 각 차원 점수가 API 응답에 포함되는지 검증한다.
- 클러스터링 기반 등급 분류가 빈 데이터, 편향 데이터, 극단값 데이터에서 안정적으로 동작하는지 검증한다.
- 설계변경 비교 분석이 변경 전/후 리스크 차이를 올바르게 계산하는지 검증한다.

### 10.3 데이터 검증

- 필수 CSV 컬럼 존재 여부
- 노드 ID 중복 여부
- 관계 endpoint 참조 무결성
- 빈 description/action 여부
- source version/hash 존재 여부

## 14. 결정 사항

- 프로젝트명 입력은 필수로 하지 않는다.
- 분석 범위는 전체 프로젝트 데이터 기준으로 한다.
- 북마크는 당분간 현 상태를 유지한다.
- 자연어 검색은 MVP에서 키워드 기반으로 유지한다.
- 장비 필터의 Strategy 경유 2-hop 방식은 유지한다.
- 분석 이력은 모든 사용자에게 공유되는 전역 이력으로 둔다.
- 과거 분석 재실행은 최신 데이터 기준으로 수행한다.
- 인증은 구현하지 않는다. 사용자 로그인과 API Key를 모두 사용하지 않는다.
- PRD는 루트 `PRD.md`에 둔다.
